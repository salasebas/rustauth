## Tracing today

Integrators typically initialize `tracing-subscriber` in the Axum/binary entrypoint and rely on:

* Application hooks ([`GlobalHooksOptions`](/docs/concepts/hooks)) — log or trace around each auth route.
* Plugin and adapter logs — enable `RUST_LOG=rustauth=debug,rustauth_core=debug`.
* Backend reference pattern — `tracing::debug!(path, "auth request")` in `examples/backend-reference/src/auth/options.rs`.

```rust
use tracing_subscriber::EnvFilter;

tracing_subscriber::fmt()
    .with_env_filter(EnvFilter::from_default_env())
    .init();
```

Mount RustAuth on Axum and propagate trace contexts with your framework's middleware (`tower-http` trace layers, `axum-tracing-opentelemetry`, etc.).

## HTTP error and latency signals

| Signal                  | Source                                                                        |
| ----------------------- | ----------------------------------------------------------------------------- |
| Request path / status   | Axum `TraceLayer` or reverse proxy access logs                                |
| Auth API error codes    | Parse [`ApiErrorResponse`](/docs/concepts/api#error-handling) bodies          |
| OAuth redirect failures | Parse `?error=` on redirect URLs ([errors reference](/docs/reference/errors)) |
| Rate limits             | `429` responses with `X-Retry-After` header                                   |

## Database and adapter visibility

SQLx and custom adapters should log slow queries at the integration layer. RustAuth core does not yet wrap each adapter call in OTEL `db.*` spans.

## Planned OpenTelemetry parity

Future work aims to cover:

* Endpoint spans (`{METHOD} {route}`)
* Global and plugin hook spans
* Adapter operation spans

Until then, treat upstream [Better Auth instrumentation docs](https://www.better-auth.com/docs/reference/instrumentation) as the behavioral target, not the current Rust implementation.

title: Axum Integration
description: Mount RustAuth in an Axum application with rustauth-axum.

RustAuth ships an official Axum adapter in the `rustauth-axum` crate. Use it when your server is built with Axum and you want auth routes under a path such as `/api/auth`.

<Callout type="info">
  RustAuth also supports [Actix Web](/docs/integrations/actix-web). For other HTTP stacks, use the low-level [`handle`](https://docs.rs/rustauth-axum/latest/rustauth_axum/fn.handle.html) escape hatch or call `handler_async` directly.
</Callout>

## Install

```toml
[dependencies]
rustauth = "0.1"
rustauth-axum = "0.1"
axum = "0.8"
tokio = { version = "1", features = ["macros", "rt-multi-thread", "net"] }
```

See [installation](/docs/installation) for secrets, database adapters, and plugins.

## Quick start

```rust
use rustauth::prelude::*;
use rustauth_axum::{RustAuthAxumExt, RustAuthAxumOptions};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let auth = RustAuth::builder()
        .secret("secret-a-at-least-32-chars-long!!")
        .base_url("https://app.example.com/api/auth")
        .build()
        .await?;

    let app = auth.mount_at_base_path(
        RustAuthAxumOptions::new().body_limit(1024 * 1024),
    )?;
    // serve with ConnectInfo when clients hit Axum directly (rate limiting IP)
    Ok(())
}
```

Run `rustauth db migrate --yes` before serving traffic when using a database adapter. See [database migrations](/docs/database-migrations).

When Axum is exposed directly to clients, serve with connection info so RustAuth can use the real peer socket IP for rate limiting:

```rust
use std::net::SocketAddr;
use axum::serve;
use tokio::net::TcpListener;

async fn run(app: axum::Router) -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:3000").await?;
    serve(
        listener,
        app.into_make_service_with_connect_info::<SocketAddr>(),
    )
    .await?;
    Ok(())
}
```

## Choosing a mount API

| Goal                                | API                                                                  | Notes                |
| ----------------------------------- | -------------------------------------------------------------------- | -------------------- |
| Standalone auth server              | `auth.mount_at_base_path(RustAuthAxumOptions::default())`            | Nests at `base_path` |
| Shared `Arc<RustAuth>` in app state | `auth.mount_routes(RustAuthAxumOptions::default())` + your `.nest()` | You control prefix   |
| Let rustauth-axum nest              | `auth.mount_at_base_path(RustAuthAxumOptions::default())`            | Do not nest again    |
| Custom state on auth routes         | `mount_routes(options)`                                              | Same as above        |

When you nest manually with `mount_routes()`, ensure your outer `.nest(prefix, …)` uses the same path as `RustAuthOptions::base_path` and that `base_url` matches.

## Production notes

* Default mount path comes from `RustAuthOptions::base_path`, falling back to `/api/auth`.
* Request bodies are collected before core and capped at 10 MiB by default.
* Configure `RustAuthOptions::base_url` in production. Do not trust public `x-forwarded-for` unless traffic is terminated by a trusted reverse proxy.
* Do not run Tower/Axum body-consuming middleware on auth routes before `rustauth-axum`.

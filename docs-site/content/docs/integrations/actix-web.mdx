RustAuth ships an official Actix Web adapter in the `rustauth-actix-web` crate. Use it when your server is built with Actix Web and you want auth routes under a path such as `/api/auth`.

<Callout type="info">
  RustAuth also supports [Axum](/docs/integrations/axum). For other HTTP stacks, use the low-level [`handle`](https://docs.rs/rustauth-actix-web/latest/rustauth_actix_web/fn.handle.html) escape hatch or call `handler_async` directly.
</Callout>

## Install

```toml
[dependencies]
rustauth = "0.1"
rustauth-actix-web = "0.1"
actix-web = "4"
```

See [installation](/docs/installation) for secrets, database adapters, and plugins.

## Quick start

```rust
use std::sync::Arc;

use actix_web::{App, HttpServer};
use rustauth::prelude::*;
use rustauth_actix_web::{RustAuthActixWebExt, RustAuthActixWebOptions};

#[actix_web::main]
async fn main() -> std::io::Result<()> {
    let auth = Arc::new(
        RustAuth::builder()
            .secret("secret-a-at-least-32-chars-long!!")
            .base_url("https://app.example.com/api/auth")
            .build()
            .await
            .expect("valid RustAuth config"),
    );

    HttpServer::new(move || {
        App::new().service(
            auth.mount_at_base_path(RustAuthActixWebOptions::default())
                .expect("valid RustAuth Actix mount"),
        )
    })
    .bind(("127.0.0.1", 8080))?
    .run()
    .await
}
```

Run `rustauth db migrate --yes` before serving traffic when using a database adapter. See [database migrations](/docs/database-migrations).

When Actix Web is exposed directly to clients, ensure the server records peer socket addresses so RustAuth can use the real client IP for rate limiting. The adapter reads `HttpRequest::peer_addr()` when `use_peer_addr_for_ip` is enabled (the default).

## Choosing a mount API

| Goal                                | API                                                                              | Notes                |
| ----------------------------------- | -------------------------------------------------------------------------------- | -------------------- |
| Standalone auth server              | `auth.mount_at_base_path(RustAuthActixWebOptions::default())`                    | Nests at `base_path` |
| Shared `Arc<RustAuth>` in app state | `auth.mount_routes(RustAuthActixWebOptions::default())` + your `.service()` nest | You control prefix   |
| Let rustauth-actix-web nest         | `auth.mount_at_base_path(RustAuthActixWebOptions::default())`                    | Do not nest again    |
| Custom state on auth routes         | `mount_routes(options)`                                                          | Same as above        |

When you nest manually with `mount_routes()`, ensure your outer scope uses the same path as `RustAuthOptions::base_path` and that `base_url` matches.

## Production notes

* Default mount path comes from `RustAuthOptions::base_path`, falling back to `/api/auth`.
* Default request body limit is 10 MiB; oversized bodies return JSON `413 PAYLOAD_TOO_LARGE`.
* Configure `RustAuthOptions::base_url` in production. Do not trust public `x-forwarded-for` unless traffic is terminated by a trusted reverse proxy.
* Do not run body-consuming Actix middleware on auth routes before `rustauth-actix-web`.

## Actix adapter limitations

Compared to `rustauth-axum`, this adapter intentionally does not preserve:

* **Response HTTP version** — Actix `HttpResponse` has no safe response-version setter.
* **Opaque response extensions** — custom `ApiResponse` extension types are not round-tripped through Actix.
* **Arbitrary request extensions** — only RustAuth-known types (`RequestClientIp`, `RequestBaseUrl`, `OAuthBaseUrlOverride`) are copied from `HttpRequest`.

See the crate [README](https://github.com/salasebas/rustauth/tree/main/crates/rustauth-actix-web) and [UPSTREAM.md](https://github.com/salasebas/rustauth/tree/main/crates/rustauth-actix-web/UPSTREAM.md) for details.

---
title: SCIM
description: SCIM 2.0 user and group provisioning for RustAuth server plugin.
---

The SCIM plugin lets external identity providers provision users and groups into RustAuth over SCIM 2.0. It is server-side only — no browser SDK or dashboard UI.

Management routes use regular RustAuth JSON. Protocol routes use `application/scim+json` and RFC 7644 error shapes.

## Install

```toml
[dependencies]
rustauth = { version = "0.1", features = ["scim"] }
```

Your HTTP stack must allow `GET`, `POST`, `PUT`, `PATCH`, and `DELETE` on SCIM routes.

## Enable

```rust
use rustauth::RustAuth;
use rustauth::scim::{scim, ScimOptions, ScimTokenStorage};

let auth = RustAuth::builder()
    .secret("secret-a-at-least-32-chars-long!!")
    .base_url("https://app.example.com/api/auth")
    .plugin(
        scim(
            ScimOptions::default()
                .token_storage(ScimTokenStorage::Hashed),
        ),
    )
    .build()
    .await?;
```

SCIM clients call routes under `{base_url}/scim/v2` (for example `https://app.example.com/api/auth/scim/v2`).

Group provisioning maps to [organization](/docs/plugins/organization) teams — enable the organization plugin for org-scoped groups and tokens.

## Configuration

| Field | Description |
| --- | --- |
| `provider_ownership` | Allow global (non-org) provider connections |
| `required_role` | Org role required to manage org-scoped tokens |
| `default_scim` | Seed provider connections at startup |
| `token_storage` | `Hashed` (default), `Plain`, `Encrypted`, or custom |
| `before_token_generated` / `after_token_generated` | Token lifecycle hooks |
| `bulk_mode` | `Independent` (default) or `Atomic` bulk processing |
| `deprovision_mode` | `UnlinkAccount` (default) or `DeleteUser` |
| `audit_event` | Structured audit callback |

**Provider IDs:** `providerId` is globally unique — one id equals one SCIM connection. Use distinct ids per org or environment (`okta-prod`, `okta-eu`). Regenerating a token for the same id upserts the existing row.

**Token storage:** Default `ScimTokenStorage::Hashed` stores SHA-256 hashes. Use `Plain` only for local development. After switching from plain to hashed, regenerate every connection via `POST /scim/generate-token`.

**Bulk mode:** `Atomic` requires a database adapter with transaction support (SQLite/Postgres/MySQL adapters). The in-memory adapter does not qualify.

## HTTP endpoints

### Management (session-authenticated)

| Method | Path | Description |
| --- | --- | --- |
| POST | `/scim/generate-token` | Create or rotate a provider bearer token |
| GET | `/scim/list-provider-connections` | List connections |
| GET | `/scim/get-provider-connection` | Get connection by `providerId` |
| POST | `/scim/delete-provider-connection` | Delete a connection |

Example generate-token body:

```json
{
  "providerId": "okta",
  "organizationId": "org_123"
}
```

`organizationId` is optional. When omitted, set `provider_ownership.enabled: true` so only the owning session user can manage that global connection.

### SCIM 2.0 protocol (bearer token)

| Method | Path | Description |
| --- | --- | --- |
| GET/POST | `/scim/v2/Users` | List or create users |
| GET/PUT/PATCH/DELETE | `/scim/v2/Users/:userId` | Read or mutate a user |
| POST | `/scim/v2/Users/.search` | Search users |
| GET/POST | `/scim/v2/Groups` | List or create groups |
| GET/PUT/PATCH/DELETE | `/scim/v2/Groups/:groupId` | Read or mutate a group |
| POST | `/scim/v2/Groups/.search` | Search groups |
| POST | `/scim/v2/.search` | Generic search |
| POST | `/scim/v2/Bulk` | Bulk operations |
| GET | `/scim/v2/ServiceProviderConfig` | Capability metadata |
| GET | `/scim/v2/Schemas` | Supported schemas |
| GET | `/scim/v2/ResourceTypes` | Resource types |
| GET | `/scim/v2/Me` | Returns SCIM `501` |

## Database

The plugin adds:

| Table | Purpose |
| --- | --- |
| `scim_providers` | Provider connections and hashed tokens |
| `scim_user_profiles` | External SCIM user attributes |
| `scim_group_profiles` | External group attributes linked to org teams |

Run migrations after enabling:

```bash
rustauth db migrate
```

Use a durable SQL adapter in production. The in-memory adapter suits tests only.

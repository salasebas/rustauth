---
title: API Key
description: Create, verify, and rate-limit API keys for RustAuth server plugin.
---

The API key plugin lets users or organizations create long-lived API keys, verify them on requests, and optionally treat a valid key as a session. Keys are stored hashed with optional prefix and starting-character hints for display.

## Install

```toml
[dependencies]
rustauth = { version = "0.1", features = ["plugins"] }
time = "0.3"
```

## Enable

```rust
use rustauth::RustAuth;
use rustauth::plugins::api_key::{
    api_key, ApiKeyConfiguration, ApiKeyExpirationOptions, ApiKeyOptions,
    ApiKeyRateLimitOptions, ApiKeyReference, StartingCharactersConfig,
};
use time::Duration;

let auth = RustAuth::builder()
    .secret("secret-a-at-least-32-chars-long!!")
    .plugin(
        api_key(
            ApiKeyOptions::builder()
                .configuration(ApiKeyConfiguration {
                    reference: ApiKeyReference::User,
                    rate_limit: ApiKeyRateLimitOptions {
                        enabled: true,
                        time_window: Duration::days(1),
                        max_requests: 100,
                    },
                    key_expiration: ApiKeyExpirationOptions {
                        default_expires_in: Some(Duration::days(90)),
                        min_expires_in_days: 7,
                        max_expires_in_days: 365,
                        ..ApiKeyExpirationOptions::default()
                    },
                    starting_characters: StartingCharactersConfig {
                        should_store: true,
                        characters_length: 8,
                    },
                    ..ApiKeyConfiguration::default()
                })
                .build()?,
        )?,
    )
    .build()
    .await?;
```

See [Advanced](/docs/plugins/api-key/advanced) for organization-owned keys, secondary storage, and multiple configurations. See [Reference](/docs/plugins/api-key/reference) for type details.

## Configuration

Top-level **`ApiKeyOptions`**:

| Field | Description |
| --- | --- |
| `configurations` | One or more `ApiKeyConfiguration` entries |
| `schema` | Table and column overrides via `ApiKeySchemaOptions` |

Each **`ApiKeyConfiguration`**:

| Field | Description |
| --- | --- |
| `config_id` | Required when using multiple configurations |
| `reference` | `User` (default) or `Organization` |
| `api_key_headers` | Header names to read keys from (default `x-api-key`) |
| `rate_limit` | Per-key rate limiting |
| `key_expiration` | Default and bounds for key expiry |
| `starting_characters` | Store/display prefix of raw key |
| `default_prefix` / `default_key_length` | Key generation defaults |
| `enable_metadata` | Allow arbitrary metadata on keys |
| `enable_session_for_api_keys` | Treat valid keys as authenticated sessions |
| `default_permissions` | Default permission set for new keys |
| `storage` | `Database` or `SecondaryStorage` lookup mode |

Organization-owned keys require the [organization](/docs/plugins/organization) plugin.

## HTTP endpoints

Clients call these routes under your configured `base_path`.

| Method | Path | Description |
| --- | --- | --- |
| POST | `/api-key/create` | Create an API key (returns raw key once) |
| POST | `/api-key/verify` | Verify a key |
| GET | `/api-key/get` | Get key metadata by ID |
| POST | `/api-key/update` | Update key settings |
| POST | `/api-key/delete` | Delete a key |
| GET | `/api-key/list` | List keys for the current user or org |
| POST | `/api-key/delete-all-expired-api-keys` | Purge expired keys |

Protected routes can also accept keys via configured headers when session mode or custom validators are enabled.

## Database

The plugin adds an **`api_keys`** table with fields for name, hashed key, prefix, reference, rate-limit counters, expiration, permissions, and metadata.

Run migrations after enabling:

```bash
rustauth db migrate
```
